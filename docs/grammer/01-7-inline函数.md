## inline 函数（带原理版）

## 为什么用 inline

函数调用有额外开销（传参、保存/恢复现场、返回跳转）。 对“非常短且高频”的函数，这些开销可能占比明显。`inline` 的目标是让编译器在调用点直接展开函数体，减少调用开销。

## 先理解函数调用栈

CPU 执行到函数调用点时，通常会发生：

1. 准备调用环境：传递参数、保存返回地址、必要时保存寄存器等。
2. 跳转到被调函数执行函数体。
3. 函数结束后恢复现场，返回调用点继续执行。

这套“进入函数 -> 执行 -> 返回”的流程就是函数调用机制。如果函数体很短（例如 getter、简单判断），调用机制本身的成本会显得更突出。

## inline 的本质

- `inline` 是**建议**，不是强制命令。
- 是否真正内联展开由编译器决定（与优化级别、函数复杂度等有关）。
- 即使未展开，`inline` 函数仍可像普通函数一样被调用。
- 从语言规则角度看，`inline` 还影响**链接语义**：允许同一个函数定义出现在多个翻译单元中（前提是定义一致，通常放头文件实现）。


## 什么时候适合 inline

- 函数很短（通常 1~几行逻辑）
- 调用频繁
- 无复杂循环/分支，代码稳定

不适合：

- 函数体较大或逻辑复杂（可能导致代码膨胀）

## 代价与误区

- 内联可能带来代码复制，增大二进制体积（I-cache 压力可能上升）。调用多少处，就有多少份内联的二进制。
- “写了 inline 一定会内联”是误区。
- Inline函数内联后 存在于可执行文件中。
- 即便编译器没有做“性能上的内联展开”，`inline` 在多文件工程里仍有价值（避免重复定义的链接问题）。


## 多文件工程最佳实践

把 `inline` 函数的**定义**放到头文件中（让每个使用它的翻译单元都能看到定义）。

KAQ: 为什么必须放头文件:

- `inline` 函数常被多个 `.cpp` 包含和使用。
- 如果只在某个 `.cpp` 里定义，其他翻译单元看不到它的定义，容易导致链接问题或无法按预期内联。
- 放在头文件可让每个翻译单元拿到同一份定义，满足 ODR（One Definition Rule）要求。


## 关键点速记

- `inline`：请求编译器考虑内联展开。
- 是否展开：编译器决定。
- 主要收益：减少短小函数的调用开销。
- 主要风险：二进制代码膨胀。
- 最佳实践：`inline` 定义放头文件。


## static inline

两个不同维度的问题 ，static 表示把可见性缩到本文件，不希望污染全局符号命名空间。
